# Hindi CSM-1B LoRA Fine-Tuning

Fine-tuning **Sesame CSM-1B** for Hindi text-to-speech using **Unsloth**, **LoRA**, and the Hindi split of **IndicTTS**.

This repository records the complete experiment from **v1.1** through **v1.3**: the original base-versus-fine-tuned comparison, extended female-only specialization, decoding-parameter search, human listening evaluation, ASR-based WER/CER analysis, and word-level error investigation.

> **Recommended way to review the project:** read the version notes, open the validation-loss graphs, and then listen to the five fixed Hindi sentences in each comparison table. WAV links open directly in the browser on GitHub.

## Final result

The final perceptual winner was:

- **Checkpoint:** female-specialization step **2,750**
- **Decoding:** **greedy**
- **Recorded seed:** **5,107**
- **Result:** all five sentences were almost fully understandable; most samples contained only one or two mispronounced words

Checkpoint 3,000 achieved the lowest validation loss, while an ASR-based evaluation selected checkpoint 2,500 with focused sampling. Manual listening still preferred checkpoint 2,750. This disagreement is an important result: **training loss, ASR intelligibility proxies, and human-perceived quality do not measure exactly the same thing.**

# Version 1.1 — Base, mixed 500, and female 1,500

## Objective

v1.1 established the baseline pipeline and tested whether LoRA fine-tuning improved Hindi pronunciation over the original CSM-1B model.

Three stages were compared using the same female reference audio and the same five target sentences:

1. Base `unsloth/csm-1b`
2. Balanced mixed-speaker LoRA trained for 500 cumulative steps
3. Female-only specialization continued to 1,500 steps

## Training design

### Mixed-speaker stage

- Pilot dataset: **240 clips** — 120 female and 120 male
- Pilot split: **216 train / 24 validation**
- Expanded dataset: **1,200 clips** — 600 female and 600 male
- Expanded split: **1,080 train / 120 validation**
- Training schedule: **100 pilot steps + 400 expanded-data steps**
- Cumulative mixed checkpoint: **500 steps**

### Female-specialization stage

- Valid female recordings after filtering: **2,831**
- Split: **2,689 train / 142 validation**
- Audio duration filter: **1.5–8.0 seconds**
- Sample rate: **24 kHz**
- Female checkpoint evaluated in v1.1: **1,500 steps**

## v1.2 perceptual findings

- **1,750:** complete target Hindi words became clearly recognizable for the first time.
- **2,000–2,250:** pronunciation and target-word recognition continued to improve.
- **2,500–2,750:** several samples produced smoother, clearer sentence fragments.
- **3,000:** strongest overall result across most samples, with multiple correct target words and occasional near-correct sentence fragments. Some short sections still contained fumbling or mispronunciations.
- **Important observation:** quality remained sample-dependent. Individual outputs from 2,500 or 2,750 occasionally sounded smoother than 3,000 despite the latter having the lowest validation loss.

## v1.2 conclusion

Female-only specialization produced a clear progressive improvement over the base and mixed-speaker stages. Step 3,000 was initially treated as the strongest overall checkpoint, but the sample-dependent differences motivated a more systematic decoding and checkpoint search in v1.3.

# Version 1.3 — Decoding sweep and automated evaluation

## Objective

v1.3 tested whether inference settings could improve pronunciation, pacing, and stability without additional training. It also introduced automated Hindi intelligibility analysis to complement human listening.

## Inference sweep

Three checkpoints were tested:

- 2,500
- 2,750
- 3,000

Three decoding configurations were tested:

| Configuration | Main settings |
|---|---|
| `greedy` | Sampling disabled for both main and depth decoders |
| `focused` | Temperature 0.5, top-k 20, top-p 0.90 |
| `moderate` | Temperature 0.7, top-k 50, top-p 0.90 |

Three recorded seeds were used: `3407`, `4107`, and `5107`.

Total outputs:

```text
3 checkpoints × 3 decoding configurations × 3 seeds × 5 sentences = 135 WAV files
```

> With greedy decoding, the seed is not expected to be a meaningful sampling hyperparameter because sampling is disabled. It is retained in filenames and manifests for reproducibility. Small perceived differences can still arise from runtime nondeterminism or subjective listening.

## ASR-based Hindi intelligibility evaluation

All 135 sweep outputs were transcribed with `openai/whisper-large-v3-turbo`. The normalized predictions were compared with the target sentences using:

- Word Error Rate (WER)
- Character Error Rate (CER)
- Word-level dynamic-programming alignment

Lower WER and CER indicate better agreement with the independent ASR model, but these scores remain **proxy metrics**. Whisper can make its own recognition errors and therefore cannot replace direct human listening.

### ASR result

The automatic ASR ranking selected:

- **Checkpoint:** 2,500
- **Configuration:** focused
- **Seed:** 4,107
- **Mean WER:** 0.5852
- **Mean CER:** 0.3335

![Top inference configurations by ASR-based Hindi intelligibility](metrics/asr_intelligibility_ranking.png)

### Manual winner ASR transcript examples

For the manually selected 2,750 greedy system, Whisper still produced substantial errors, including examples such as:

- `धन्यवाद` → `तन्न आद`
- `शुभ हो` → `शाल है`
- `ऑर्डर नंबर` → highly distorted ASR text

This showed that some disagreement came from TTS pronunciation errors, while some likely came from ASR recognition errors. A WER above 1.0 is valid when substitutions, deletions, and insertions together exceed the number of reference words.

## Word-level failure analysis

Target and predicted words were aligned using dynamic programming and marked as:

- correct
- substituted
- deleted
- inserted

The analysis helps locate repeatedly difficult words or sentence regions. It should be interpreted as an **ASR-derived diagnostic**, not direct phonetic ground truth.

## Comparison of selection methods

| Selection method | Selected system | Interpretation |
|---|---|---|
| Lowest validation loss | Step 3,000 | Best optimization metric, but not the best perceived output |
| Lowest ASR-derived WER/CER | Step 2,500, focused, seed 4,107 | Best Whisper agreement, but not preferred during listening |
| Manual perceptual evaluation | Step 2,750, greedy, seed 5,107 | Best overall pronunciation, intelligibility, stability, and naturalness |

## v1.3 conclusion

Checkpoint 2,750 with greedy decoding and seed 5,107 was selected as the final configuration. Across all five target sentences, speech was almost completely understandable, with only one or two mispronounced words in most samples.

The central experimental finding is:

> Validation loss selected step 3,000, ASR-based scoring selected step 2,500, and human listening selected step 2,750. These criteria measure different aspects of TTS quality, so final checkpoint selection should combine quantitative diagnostics with perceptual evaluation.

# Engineering decisions

## Fixed reference conditioning

Every model stage used the same clean female reference clip and exact reference transcript. This reduced variation caused by speaker prompting and made checkpoint comparisons more meaningful.

## Balanced mixed-speaker warm-up

The first 500 steps used balanced female/male subsets to test broad Hindi adaptation before specializing the model to the female speaker distribution.

## Female-only specialization

The full valid female subset was then used because the required comparison used a fixed female reference and female outputs showed stronger target alignment.

## Audio and text filtering

Examples were filtered for usable duration, finite waveform values, minimum signal level, and reasonable transcript length. Hindi text was normalized using Unicode NFC, whitespace cleanup, and repeated punctuation cleanup.

## Human evaluation remained primary

ASR metrics were added to improve repeatability, but human listening remained the final selection criterion because automatic transcriptions contained obvious recognition errors.

# Challenges and fixes
Google Colab runtime limitations

The assignment requested a Google Colab-compatible solution. Initial experiments were started on Colab, but the available free GPU sessions were unstable and frequently disconnected long training runs. The account also had no remaining guaranteed compute units, which made uninterrupted multi-stage training unreliable.

To avoid losing checkpoints and generated outputs, the complete training and evaluation pipeline was finished on Kaggle. The project still uses standard Python, PyTorch, Hugging Face, Unsloth, and PEFT components, so the notebook is designed to remain compatible with a sufficiently provisioned Colab GPU runtime.

Checkpoint recovery, output reuse, and session-restoration logic were added so interrupted runs could continue without restarting the full experiment.

## CSM padding-token compatibility

The installed CSM/Unsloth stack required a dedicated padding token instead of reusing EOS. The local processor used:

```text
<|finetune_right_pad_id|>
```

with token ID `128004`, while EOS remained `128001`. Local Unsloth padding helper functions were patched before loading the model.

## Recovery input format

Kaggle mounted the recovery artifact as a directory rather than a ZIP file. The restoration logic was changed to detect whether the input was a directory or archive and then use `copytree` or archive extraction accordingly.

## GPU memory management

Models were explicitly deleted between checkpoint evaluations, followed by garbage collection and CUDA cache clearing. This allowed multiple adapters and Whisper evaluation to run in one notebook session.

## Validation loss did not perfectly track perceived quality

Later checkpoints generally had lower validation loss, but some earlier samples sounded smoother. This motivated the v1.3 sweep and multi-method evaluation.

# Limitations

- Only five fixed sentences were used for the primary comparison.
- Manual listening was performed by a limited number of listeners.
- Whisper WER/CER can reflect both TTS errors and ASR errors.
- The female subset corresponds to a narrow speaker distribution, so the final adapter is specialized rather than broadly multi-speaker.
- Greedy outputs should be largely deterministic; seed-labelled greedy comparisons should not be interpreted as conventional sampling-seed experiments.

# Future improvements

- Conduct a blind listening test with multiple independent listeners.
- Add mean-opinion-score-style ratings for intelligibility, naturalness, pronunciation, and speaker consistency.
- Evaluate on a larger and more linguistically diverse Hindi test set.
- Add phoneme-aware analysis for aspiration, retroflexion, vowel length, and conjunct consonants.
- Test more reference clips and quantify speaker-reference sensitivity.
- Compare the specialized adapter with a clean uninterrupted 3,000-step run.
- Package a lightweight inference script and model card for deployment.

# Final conclusion

The project progressed from a basic base-versus-LoRA comparison in v1.1 to an extended checkpoint study in v1.2 and a systematic decoding and automated-evaluation study in v1.3. Female-only specialization substantially improved Hindi pronunciation and target alignment. The final selected system—checkpoint 2,750 with greedy decoding—produced the strongest overall perceptual quality across the five required sentences, even though other checkpoints were preferred by validation loss and ASR-based scoring.
